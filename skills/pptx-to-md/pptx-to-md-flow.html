pptx-to-md 处理流程

PPTX → PDF → 委托 pdf-to-md · 本 Skill 是薄包装层 · 更新至当前版本

💡 核心架构：薄包装层 + 能力复用
本 Skill 不再实现自己的 PNG 渲染 / OCR / Vision 流程。 所有文字提取、嵌入图保存、Vision 描述逻辑统一由 pdf-to-md 维护—— pptx-to-md 只负责 PPTX → PDF 这一步，之后整批委托给 pdf-to-md。 pdf-to-md 的任何优化（如未来的视觉密集页智能分流）自动惠及 PPTX。

所有 PPTX 转完
2
Phase 2 · 委托 pdf-to-md  一次性批处理
subprocess 调用 pdf_to_md.py --input <pdf_dir> --output <output_dir>，所有参数透传
delegate ▼
sys.executable \
  ~/.cc-switch/skills/pdf-to-md/scripts/pdf_to_md.py \
  --input <pdf_dir> --output <output_dir> \
  --large-image-kb 30 --max-images 50 \
  [--no-images] [--api-key …] [--model …]
为什么用一次性批处理而不是逐文件调用？
pdf_to_md.py 接受目录作为 input，会自动遍历所有 PDF。一次性调用避免 为每个 PPTX 单独 fork Python 进程，减少 pymupdf 重复加载开销。
pdf-to-md 内部流程（不在本 Skill 范围）：
· is_text_pdf() 判断原生文字 vs 扫描（LibreOffice 转出的通常是原生文字 PDF）
· pymupdf 直接提取文字（保留标题层级，按 font-size 启发式）
· 嵌入大图（≥ large-image-kb）保存到 <stem>/imgs/
· 文字中插入 ![](…) 占位符
· Agent 模式：返回 Agent 用 Vision 填写占位符
· Standalone 模式：脚本内部 call_vision() 自行填写
完成
3
填写图片描述  Agent 模式 · 有嵌入图时执行
由调用方 agent 完成（pdf-to-md SKILL.md Step 1b）：Read 图片 → Edit 替换占位符

≤ 5 张图片
直接用 Read tool 逐张读取
Edit 替换占位符
> 5 张 / 多文件批量
spawn subagents（每个 10–20 张）
图片字节只进 subagent 上下文
本步骤逻辑完全复用 pdf-to-md 的 Step 1b，无需在 pptx-to-md SKILL.md 中重复说明

⚠ 已知简化（与旧版差异）：
旧版本对每张幻灯片单独渲染 PNG → OCR → Vision，能精确还原视觉布局（流程图箭头、左右对比、架构图连接线）。 本版本走 PDF 文字路径，PPTX 中通过空间布局表达的信息会变成线性阅读顺序，会丢失部分视觉关系。

未来路径：如果需要精确还原视觉结构，应在 pdf-to-md 端 增加"视觉密集页"识别 + Vision 整页描述的分流逻辑——一处优化，pdf/pptx 都受益。 本 Skill 不再重复实现该能力。
✅ 最终输出结构（与 pdf-to-md 一致）
output_dir/
  ├── 文件A.md              ← 文字 + 图片占位符
  └── 文件A/
      └── imgs/
          ├── img_001.png  ← 嵌入大图（持久保存）
          └── …
· 中间 PDF：默认 tempdir 处理完删除；--pdf-dir 可持久化便于排查
· 嵌入图持久保存在 <stem>/imgs/，agent 可在任意时间点 Read
· PPTX → PDF 失败：写入 *[PDF conversion failed]* 占位 .md，不阻断其他文件
· 已移除：per_page 桩文件、slide_NNN.png 渲染、PaddleOCR 调用、--merge-only 模式（统统不再需要）
