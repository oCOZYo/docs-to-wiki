Stage 0 · 确认参数与凭证
确认 SOURCE_DIR、COMPANY_NAME、WIKI_DIR、WORKERS；检查 API Key
交互 ▼
参数说明
SOURCE_DIR包含所有文档的根目录
COMPANY_NAMEWiki 显示名称（如"数美科技"）
WIKI_DIRWiki 输出目录（默认 SOURCE_DIR/../<slug>_wiki）
WORKERS并行 worker 数，默认 4（用于 LibreOffice + OCR）
凭证检查：
· ANTHROPIC_API_KEY — Vision DOCX/PPTX 提取必需
· PADDLEOCR_TOKEN / PADDLEOCR_API_URL — 扫描 PDF OCR 必需
· DOCS_TO_WIKI_MODEL（可选）— 统一覆盖 Vision 模型；未设置时询问用户

Stage 1 · 收集文档
01_collect_docs.py — 扫描并归集 PPTX/PDF/DOCX，生成 staging 目录
I/O ▼
python ~/.claude/skills/docs-to-wiki/scripts/01_collect_docs.py \
  --source SOURCE_DIR \
  --output SOURCE_DIR/../<slug>_staging
收集跳过
PPTX、PPSX、PDF、DOCX Excel/CSV、隐藏文件、临时文件（~$ 前缀）、wiki/OCR 输出目录本身
输出：扁平 staging 目录，文件名中路径分隔符替换为 __（sanitized filenames）
报告：收集总数 + 格式分类统计
Stage 2 格式转换（2a / 2b / 2c 可并行） 委托三个原子 Skill
2a DOCX → MD
委托 docx-to-md skill
python-docx 无损提取 + Vision 图片描述
export ANTHROPIC_API_KEY="..."
~/.venvs/paddleocr/bin/python \
  ~/.cc-switch/skills/docx-to-md/scripts/docx_to_md.py \
  --input SOURCE_DIR \
  --output SOURCE_DIR/../<slug>_sources_tmp/ \
  --large-image-kb 30 \
  --model claude-haiku-4-5-20251001
运行后，将每个 .md 移入 per-doc 目录：
<slug>_sources/<docname>/<docname>.md
2b PPTX → MD
委托 pptx-to-md 薄包装层
LibreOffice 转 PDF → 再交 pdf-to-md
export ANTHROPIC_API_KEY="..."
~/.venvs/general/bin/python \
  ~/.cc-switch/skills/pptx-to-md/scripts/pptx_to_md.py \
  --input SOURCE_DIR \
  --output SOURCE_DIR/../<slug>_sources_tmp/ \
  --large-image-kb 30 \
  --model claude-haiku-4-5-20251001
Fallback（无 Vision / 控制成本）：
02_convert_to_pdf.py --source SOURCE_DIR --output <slug>_pdf --workers WORKERS
仅转 PDF，交给 Stage 3 OCR
2c PDF → MD
委托 pdf-to-md skill (Step 1)
原生文字 PDF，扫描 PDF 跳过留 Stage 3
~/.venvs/paddleocr/bin/python \
  ~/.cc-switch/skills/pdf-to-md/scripts/pdf_to_md.py \
  --input SOURCE_DIR \
  --output SOURCE_DIR/../<slug>_sources_tmp/ \
  --large-image-kb 30
原生 PDF avg >50 ch/pg → 直接生成 .md
扫描 PDF avg ≤50 ch/pg → 跳过，进 Stage 3
3
Stage 3 · 扫描 PDF OCR（并行批处理）
03_run_ocr.py — 并行调用 pdf-to-md Step 2（ocr_extract.py），批量处理失败自动重试
PaddleOCR 并行 ▼
原子 Skill 扩展：ocr_extract.py 本身顺序执行，docs-to-wiki 在外层加并行批处理 wrapper，大幅提速大规模语料
export PADDLEOCR_TOKEN="..."
export PADDLEOCR_API_URL="..."
python ~/.claude/skills/docs-to-wiki/scripts/03_run_ocr.py \
  --pdf-dir SOURCE_DIR/../<slug>_pdf \
  --original-pdf-dir SOURCE_DIR \
  --output SOURCE_DIR/../<slug>_ocr \
  --workers WORKERS --batch-size 25
OCR 重试（Stage 3†）：网络中断后，重复运行以下命令直到 remaining count 稳定：
python ... 04_run_ocr_remaining.py --pdf-dir ... --ocr-output ... --workers WORKERS --batch-size 25
特性说明
内部调用~/.cc-switch/skills/pdf-to-md/scripts/ocr_extract.py（并行批次）
去重按文件名 stem 去重，避免重复 OCR
幂等性可安全多次运行，retry 脚本检查已有输出

Stage 4 · 合并 OCR 输出
05_merge_ocr.py — 整合 batch/remaining 子目录 + Stages 2a/2b/2c 产出，生成统一 sources 结构
I/O ▼
python ~/.claude/skills/docs-to-wiki/scripts/05_merge_ocr.py \
  --ocr-dir SOURCE_DIR/../<slug>_ocr \
  --output SOURCE_DIR/../<slug>_sources
输出目录结构：
<slug>_sources/
  <docname>/
    <docname>.md     ← 最终 MD 文件
    imgs/           ← 提取图片（仅 OCR 路径）
5
Stage 5 · 构建 Wiki（LLM 合成）
核心合成阶段：分类设计 → 源文件映射 → 并行 subagent 写 Wiki 页
并行 Agent ▼
开始前，先读 references/wiki-build-guide.md 获取完整指引
步骤操作
① 采样随机读取约 20 个源文档名，理解领域
② 分类设计设计 Wiki 类目，编写 classify() 函数
③ 源文件映射构建 source map（文档 → 类目映射）
④ 初始化结构创建 CLAUDE.md、index.md、log.md
⑤ 并行写页每类目一个 subagent，读完源文件立即写页面（不批量读后再写）
⑥ 移入 sources将 <slug>_sources/ 移入 wiki/sources/
⑦ 重建链接运行 Stage 6 脚本
重要约束：Wiki 页面只含文字合成，禁止添加批量图片 Gallery——读者通过 sources/ 链接查看带图的原文
6
Stage 6 · 重建 Wikilink
06_rebuild_source_links.py — 将纯文字来源替换为 [[sources/xxx|display]] wikilink
pyyaml ▼
~/.venvs/paddleocr/bin/python \
  ~/.claude/skills/docs-to-wiki/scripts/06_rebuild_source_links.py \
  --wiki-dir WIKI_DIR
读取每个 Wiki 页面的 YAML frontmatter sources: 列表， 将 ## 来源 节替换为 [[sources/xxx|display_name]] wikilink。 需要 pyyaml（~/.venvs/paddleocr 已包含）。
7
Stage 7 · Pipeline 报告
写入 WIKI_DIR/pipeline_report.md — 覆盖完整转换统计
报告 ▼
报告章节内容
Raw 文件收集按格式分类（PPTX/PDF/DOCX/…）
格式转换按原子 Skill 统计（docx-to-md / pptx-to-md / pdf-to-md 文字路径）
OCR输入 PDF 数、成功、失败、成功率、提取图片数、大小
Wiki按类目页面数、源文件覆盖率
目录结构最终 tree
参见 references/pipeline-report-template.md 获取精确格式模板
完成
